# repo-metadata-signal-v1

## Status

Repository metadata note v1.

## Function

This document records the current **front-door metadata recommendations** for the `aminglab/knowledge-os` repository.

Its purpose is practical:

- to improve first-visit legibility,
- to reduce the chance that the repository is mistaken for a generic AI wrapper,
- and to give the maintainer exact copy for low-cost, high-signal GitHub repository settings.

This file does **not** replace repository content.
It exists because repository settings such as description and topics are part of the public-facing surface, but they do not live inside normal markdown pages.

---

## Why this matters now

The repository already has much stronger internal structure than its outer GitHub shell immediately signals.

At this stage, a stranger can already find:

- a protocol layer,
- a real living case,
- public entry surfaces,
- source pages,
- claim pages,
- a publishing pipeline,
- and a seeded public-layer ecology.

But repository-level metadata still carries a lot of first-impression weight.

That means a few very small public signals can have disproportionate value.

---

## Recommended repository description

### Short version

**A protocol and operating environment for governing living knowledge through claims, evidence, dissent, verdicts, snapshots, and lineage.**

### Slightly more product-facing version

**Knowledge OS is a protocol layer and operating environment for living knowledge: claims, evidence, dissent, verdicts, snapshots, and lineage.**

### Why this wording

The description should do three things fast:

1. say this is a protocol / operating environment,
2. say it is about living knowledge rather than generic AI tooling,
3. expose the core object vocabulary without overexplaining.

The shorter version is slightly harder and cleaner.
The second version is slightly more readable to a new visitor.

## Recommended repository front-door interpretation

A first-time visitor should be able to infer all three of the following within seconds:

1. this is not a generic AI wrapper,
2. this is not only a white paper repository,
3. this is an attempt to define and demonstrate a real protocol for governed living knowledge.

That means the repository front door should signal:

- protocol,
- governance,
- living knowledge,
- and real implementation pressure.

## Recommended minimum front-door checklist

### Already present

- root `README.md`
- `PUBLIC-ENTRY.md`
- `PUBLIC-CASES.md`
- real public case layer
- seeded claim and source layers
- LICENSE file

### Still worth setting in GitHub repository settings

- repository description
- topic tags

### Optional later signals

- social preview image
- pinned repository order at profile level
- website field only if a real external surface exists later

---

## Recommended sequencing

The practical order should be:

1. keep the current README / public entry chain,
2. set repository description,
3. set topic tags,
4. leave heavier public-profile polish for later.

This keeps the front door sharper without slipping into vanity work.
