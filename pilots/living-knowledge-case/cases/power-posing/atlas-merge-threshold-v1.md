# Atlas Merge Threshold v1

This file records the current **atlas merge threshold v1** for the `power-posing` pilot.

Its job is narrow:

> name the future threshold at which merging `check-atlas-v1.md` and `public-layer-verification-atlas-v1.md` could become admissible,
> without turning that threshold into a present-tense merge instruction.

---

## One-sentence ruling

Current ruling:

> **HOLD_NO_ATLAS_MERGE.**
> **The dual-atlas structure remains in force.**
> **This file names future merge thresholds; it does not authorize a merge now.**

---

## Why this file exists

The current project now has:

- a broader case-scoped atlas,
- a narrower public-layer atlas,
- and an explicit authority boundary ruling explaining why both currently coexist.

That already governs the present.
What was still missing was a lighter note for the future question:

> when, if ever, would it become reasonable to merge the two atlas files?

This file exists to answer that question without prematurely collapsing the current structure.

---

## Current threshold test

A merge should not be treated as admissible unless **all** of the following thresholds are met strongly enough.

### 1. Scope convergence threshold
The broader atlas and the narrower atlas must have converged enough that one document can represent both without obvious bloat or thinning.

Typical signs:

- `check-atlas-v1.md` contracts strongly toward the public-layer subset,
- or `public-layer-verification-atlas-v1.md` expands until it has become the dominant case-scoped map.

### 2. Entry-surface simplification threshold
A merged atlas must make the reading path simpler rather than more confusing.

Typical signs:

- README no longer needs a two-atlas path to stay legible,
- one first-stop atlas would clearly reduce navigation friction,
- and the authority note would shrink rather than grow after consolidation.

### 3. Boundary-preservation threshold
A merged atlas must preserve the current public-layer operational boundary without dissolving it into generic case prose.

Typical signs:

- public-layer orchestration, public-layer-only checks, and public-layer-specific gaps can still be surfaced cleanly inside a merged map,
- and the public-layer sub-map would not disappear into a broader undifferentiated checker inventory.

### 4. Duplicate-mass threshold
Enough duplicated explanatory mass must exist that consolidation produces real compression rather than pseudo-efficiency.

Typical signs:

- the two files repeatedly restate the same authority note,
- the same boundary explanations dominate both files,
- and maintaining both has become a clearer source of drift than keeping them separate.

### 5. Upstream pressure threshold
There should be a larger architectural reason to merge.

Typical signs:

- a stronger multi-case atlas program appears,
- a repository-wide atlas framework is being introduced,
- or a broader validation re-architecture forces the atlas layer to be reassembled anyway.

---

## Current verdict against the threshold

Current verdict:

- **scope convergence threshold:** not met
- **entry-surface simplification threshold:** not met
- **boundary-preservation threshold:** not met strongly enough for safe consolidation
- **duplicate-mass threshold:** not met strongly enough
- **upstream pressure threshold:** not met

So the current result remains:

> **HOLD_NO_ATLAS_MERGE.**
