# public-layer-foundation-v1

## Status

Foundation note v1.

## Function

This document defines the current **public-layer foundation** earned by the `power-posing` line at the start of Phase 3.

It is not a full product specification.
It is not a global protocol freeze.
It is not a declaration that the entire public layer has already been built.

Its job is narrower and more useful:

- to state why the public layer exists,
- to define what kind of thing it is,
- to clarify what belongs in it and what does not,
- to define how it relates to the private cockpit, team layer, snapshot layer, and page layer,
- and to explain how the first validated case surface now functions as the seed implementation for a broader public-layer direction.

---

## Why this document exists now

The repository has already crossed an important threshold.

It no longer contains only:

- founding narrative,

It now also contains:

## Core definition

The **public layer** is the release surface where selected governed knowledge objects become publicly readable, publicly navigable, and publicly citable without collapsing the deeper governed graph into a single frozen document.

It is therefore not just a publishing endpoint.
It is a structured release surface over governed objects.

The public layer exists so that public knowledge artifacts can be:

- released deliberately rather than leaked from internal workspaces,
- read as living knowledge rather than only as static documents,
- traced back to explicit governed objects,
- connected to source surfaces, claim lineage, dissent, and visible judgment,
- and revised through later release views without pretending that publication ended governance.

A short definition is:

> The public layer is the selective release surface of Knowledge OS.
> It turns governed objects into public knowledge artifacts without confusing public release with final truth.

---

## What problem the public layer solves

Knowledge OS does not attack publication as such.
It attacks the older pattern in which publication becomes the only thing the outside world can see.

In that older pattern:

- the paper survives,
- the knowledge status changes,
- and later judgment gets scattered across later papers, statements, blog posts, review threads, citations, and memory.

The public layer exists to solve that structural gap.

More concretely, it exists so that a public reader can encounter not only:

- a headline claim,
- or a polished summary,

but also some visible form of:

- current judgment,
- claim lineage,
- source grounding,
- dissent pressure,
- and release identity.

The public layer does **not** solve everything.
It does not expose the whole live graph.
It does not turn the public into a fully privileged governance participant by default.

Its role is more precise:

> to expose a governed public view of living knowledge.

---

## Public layer versus the other two product surfaces

Knowledge OS already names three connected surfaces:

- Private Cockpit
- Team Layer
- Public Layer

This document sharpens the boundary between them.

### 1. Private Cockpit

The private cockpit is where a person or small team drafts, decomposes, maps, and governs a project from the inside.

Its center is:

- exploration,
- object creation,
- route formation,
- uncertainty handling,
- and internal pressure before release.

The private cockpit is allowed to be messy in ways the public layer is not.

### 2. Team Layer

The team layer is where permissions, review, responsibility, assignment, and audit become explicit.

Its center is:

- shared governance,
- role coordination,
- review queues,
- assignment,
- accountability,
- and institutional responsibility.

The team layer is not primarily a release surface.
It is a responsibility surface.

### 3. Public Layer

The public layer is where selected governed material becomes public-facing.

Its center is:

- release identity,
- public readability,
- public navigation,
- visible current judgment,
- source exposure,
- and release continuity.

The public layer is therefore neither:

- the internal workshop,
- nor the internal responsibility board.

It is the selective public surface over work that has already passed enough internal structuring to deserve release.

---

## What the public layer is not

The public layer is not any of the following.

### 1. Not the full live graph

The public layer exposes selected public views over governed objects.
It does not expose the entire internal graph by default.

### 2. Not a generic blog or publishing skin

A public layer page is not just a prettier markdown article.
It should preserve visible relation to governed objects, source surfaces, and judgment state.

### 3. Not a static document archive

Static files may still exist inside the public layer.
But the public layer itself is not defined by staticity.
It is defined by governed release views.

### 4. Not the private cockpit rendered in public colors

The public layer should not simply dump internal tool surfaces into public view.
It needs its own release discipline and reader-facing structure.

### 5. Not a public-by-default social layer

The public layer does not imply that every object, thread, dissent, or internal route becomes open participation terrain.
Public release remains selective.

### 6. Not a proof that the whole product is already built

The existence of one public case page does not mean the public layer is finished.
It means the foundation now has to be stated honestly.

## Public layer and the release chain

At the current stage, the most important release discipline is already present in the repository:

> object layer → `snapshot-v2.md` → page layer

This discipline must remain intact.

### Object layer
The governed object files remain the deepest active source in the current case.

### Snapshot layer
`snapshot-v2.md` acts as the current public release layer and public homepage for the case.
It is not just another document.
It is the governed release view that decides what the current public case should say.

### Page layer
The page is a downstream rendering surface.
It should not invent an independent storyline that competes with the snapshot release layer.

This means the public layer is not a flat thing.
It already has an internal structure:

- governed objects,
- deliberate public release view,
- downstream public rendering.

The public layer foundation must preserve that distinction rather than collapsing it.

## Minimum public-layer capabilities now visible

The current repository does not yet contain a full public-layer product.
But it does already demonstrate the minimum capability set that the public layer must preserve.

### 1. Public homepage capability

A public reader can land on a case and understand what it is, why it matters, and what the current judgment is.

### 2. Public judgment capability

The current public surface can expose visible judgment phrases that are not merely decorative, but tied back to governed judgment language.

### 3. Public source capability

The reader can move from public summary into source surfaces and object usage.

### 4. Public object traceability

The reader can move from the release narrative into explicit object identities.

### 5. Public lineage capability

The case can show that a claim does not only become true or false, but can weaken, split, and survive in narrower descendant form.

### 6. Public release continuity

The case can be understood as a current release view over a deeper governed history, not as an isolated document.

These capabilities are the real foundation. They matter more than heavier UI ambitions at this stage.

## What belongs in the public layer

The public layer should contain public artifacts that satisfy at least one of the following functions.

### A. Public entry artifacts

These help a reader understand what system they are entering.

Examples:

- home surface,
- project-level public entry,
- case index.

### B. Public case artifacts

These make an individual case legible as living knowledge.

Examples:

- case homepage,
- case page,
- release snapshot,
- timeline view,
- claim lineage summary.

### C. Public grounding artifacts

These make the public surface traceable.

Examples:

- source pages,
- references surface,
- object links,
- status legend,
- verdict grammar bridge.

### D. Public judgment artifacts

These expose visible current standing.

Examples:

- current visible judgment,
- verdict-linked status surfaces,
- release notes that show whether a case hardened, weakened, split, or stabilized.

The public layer should therefore be understood as an ecology of release artifacts, not a single page type.

---

## What does not belong in the public layer by default

The public layer should not automatically contain:

- every draft object,
- every unresolved internal note,
- every exploratory branch,
- every internal governance discussion,
- every team review queue,
- every hidden or sensitive source,
- or every machine-generated sidecar.

Those may exist in the broader system.
They do not all belong in public release.

The public layer is a selective release surface, not a raw transparency dump.

---

## Public layer and truthfulness

One of the most dangerous mistakes at this stage would be to let the public layer become a prettier form of overclaim.

The public layer must therefore obey a strict honesty discipline.

### 1. Release identity must be explicit

A public artifact should say what kind of thing it is.
For example:

- public case homepage,
- current release view,
- downstream page surface,
- seed implementation,
- not full archive,
- not full live graph.

### 2. Current judgment must be visible without pretending finality

The public layer should expose the current standing of claims, but not pretend that a current release view is the end of history.

### 3. Public wording must remain attached to governed judgment

Reader-facing phrases should remain visibly tethered to the current status legend and verdict grammar layer.

### 4. Public polish must not erase governance traces

The public layer should become easier to read, but not at the cost of hiding where judgment came from.

In short:

> the public layer should be readable without becoming theatrically certain.
