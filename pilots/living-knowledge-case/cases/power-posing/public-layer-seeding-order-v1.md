# public-layer-seeding-order-v1

## Status

Seeding-order decision note v1.

## Function

This document decides the **next seeding order** for the public layer after:

- `phase-3-direction-decision-v1.md`
- `public-layer-foundation-v1.md`
- `public-layer-ia-v1.md`

have already established:

- that Phase 3 is a public-layer foundation phase,
- that the public layer now exists in seed form,
- and that the minimum public-layer page universe is larger than one successful case page.

The purpose of this note is simple:

> to decide which public-layer page types should be seeded next, and in what order.

This is necessary because the IA now names more page types than the repository has actually implemented as first-class public surfaces.
Without a seeding-order ruling, later work would drift between page types without a clear next target.

---

## Current public-layer reality

At the current repository stage, the public layer is not empty.
It already has a real seed.

### Already present in seed form

- **Case Page seed** — through the current `power-posing` public page
- **Reading / Lineage seed** — through `snapshot-v2.md`, reading paths, timeline, and explicit object routes

### Already partially present

- **Source Page seed** — through `references.md` and `references-metadata-v1.md`

### Defined but not yet first-class

- **Home**
- **Case Index**
- **Claim Page**

So the real question is not:

- what page types exist in theory,

but:

- which of the not-yet-first-class page types should be seeded next,
- and in what order the partially present ones should be upgraded.

---

## Seeding-order principle

The correct seeding order should not be driven by novelty alone.
It should be driven by the current structural gap of the repository.

The best next seed is the one that:

1. reduces the largest current public-layer ambiguity,
2. grows naturally from what already exists,
3. does not require premature full product build-out,
4. and improves the legibility of the whole public layer rather than only one local page.

In other words:

> seed what clarifies the layer first, not what merely adds another surface.

---

## Current gap analysis

The current repository has three important public-layer gaps.

### Gap 1. No true system-level public entry yet

The system is still entered publicly through repository-level documents rather than through a dedicated public-layer entry surface.

This means that the current first case surface risks doing too much identity work by itself.
A reader can enter the case, but the system-level public layer still lacks a first-class front door.

### Gap 2. No true public case listing layer yet

Because only one case exists, the repository can get away with treating that one case as the visible public layer.
But that is a temporary convenience, not a good public-layer structure.

Without a Case Index page type, the public layer remains implicitly case-singular.

### Gap 3. Object-aware public navigation remains uneven

The public surface already demonstrates source and claim traceability, but they are not yet represented as first-class public page types.

This means public navigation still leans heavily on the case page rather than on a more mature page ecology.

---

## Formal seeding order decision

The public-layer seeding order should currently be:

### 1. Home

### 2. Case Index

### 3. Source Page

### 4. Claim Page

This is the formal current order.

---

## Why Home comes first

### 1. Because the biggest remaining ambiguity is system-level public identity

The current `power-posing` case page already proves that one public case can exist.
What the repository still does not have is a true public entry that tells a reader:

- what Knowledge OS is,
- what kind of system they are entering,
- why they are seeing a case page,
- and how the case belongs to a larger public layer.

That is a bigger current gap than any individual missing case-adjacent page type.

### 2. Because Home reduces pressure on the first case page

Right now, the first case page does too much work:

- case introduction,
- public-layer explanation,
- and part of the system’s public identity burden.

A real Home page would remove that overload and let the case page be more clearly a case page.

### 3. Because Home does not require heavy interactivity yet

A Home seed can be small and honest.
It does not require a full application shell.
It only requires that the public-layer entry stop being implicit.

So Home is the correct first seed.

---

## Why Case Index comes second

### 1. Because the system must stop reading as if one case equals the whole public layer

Even with only one case, a Case Index page type is valuable.
It says:

- the public layer can hold multiple cases,
- the current case is the first case, not the only conceivable case,
- and public entry should not collapse directly into one case surface.

### 2. Because Case Index is the lightest next page after Home

Case Index can begin as a very small listing surface.
It does not require heavy graph navigation or object-deep rendering.
It only needs:

- case card,
- short synopsis,
- current visible judgment summary,
- and route into the case page.

That makes it a high-value, low-complexity next seed.

### 3. Because it strengthens public-layer shape without pretending scale

A one-card Case Index is still structurally useful.
It says the public layer has a listing surface, even if the list is currently short.

So Case Index should come immediately after Home.

---

## Why Source Page comes third

### 1. Because Source Page already has the strongest partial seed

The repository already has:

- `references.md`
- `references-metadata-v1.md`
- source cards on the case page,
- and explicit source roles.

So Source Page is not starting from zero.
It is already partially present.

### 2. Because it deepens public traceability without requiring full claim-graph public navigation yet

Source Page strengthens one of the most valuable things the current case already proved:

- that public living knowledge should remain traceable to visible source surfaces.

This is a lower-risk next step than Claim Page because source surfaces are already strongly legible in the current case.

### 3. Because it improves the public layer as a release ecology, not only as a narrative layer

Without first-class source surfaces, the public layer remains too summary-shaped.
A Source Page seed would push it toward being a real ecology of release artifacts.

So Source Page should come before Claim Page.

---

## Why Claim Page comes fourth

### 1. Because Claim Page is important but structurally more demanding

Claim Page is the first page type that begins to expose the public layer as object-aware rather than only case-aware.
That makes it valuable.
But it also makes it more demanding.

It needs stronger decisions about:

- current visible judgment at claim level,
- claim lineage rendering,
- support and dissent grouping,
- and public object identity without overwhelming the reader.

### 2. Because Claim Page is better seeded after system entry and source surfaces are clearer

Once Home and Case Index exist, and once Source Page is more first-class, a Claim Page seed will land in a more coherent public universe.

Without those earlier seeds, Claim Page risks arriving into a public layer that still lacks basic system-level framing.

### 3. Because Claim Page is where public-layer objecthood becomes more visible

That is precisely why it should not be rushed before the simpler structural surfaces are in place.

So Claim Page should come fourth, not because it is unimportant, but because it is the first deeper object-facing page type and should be seeded into a clearer layer.

---

## Seeding-order summary table

### Order 1 — Home
Purpose:

- establish true public entry
- clarify system identity
- separate system-level public entry from case-level entry

### Order 2 — Case Index
Purpose:

- establish public case listing surface
- stop treating one case as the whole public layer
- create a stable bridge from Home into the current first case

### Order 3 — Source Page
Purpose:

- upgrade the already-partial source surface into a first-class public page type
- strengthen public traceability and source-role visibility

### Order 4 — Claim Page
Purpose:

- expose object-aware public reading at claim level
- deepen case navigation into governed object identity

---

## What this order explicitly rejects

This seeding order rejects the following moves as current first seeds.

### 1. Do not seed Claim Page first

Doing so would deepen object-facing public reading before the public layer has a true front door.
That is the wrong order.

### 2. Do not skip Home and Case Index because only one case exists

That would keep the public layer too case-singular and would overload the first case page with system-level identity work.

### 3. Do not jump directly to a larger app shell

The current task is seeding order, not full public product construction.

### 4. Do not treat Source Page as optional decoration

Source surfaces are already one of the strongest values of the current case.
They deserve explicit public-layer upgrading.

---

## Current recommended operational sequence

If this seeding order is accepted, the next operational sequence should be:

### Step 1. Home seed planning

Define what the smallest honest public entry surface should contain.

### Step 2. Case Index seed planning

Define how the first case should appear as one item in a public-layer list rather than as the whole visible system.

### Step 3. Source Page seed planning

Decide how canonical sources should become first-class public surfaces.

### Step 4. Claim Page seed planning

Only after the previous three steps should the public layer decide how to expose a deeper claim-centered page type.

---

## Current stage verdict

Current verdict:

> The next public-layer seed should not be chosen by what looks most sophisticated.
>
> It should be chosen by what best clarifies the public layer as a system.
>
> At the current repository stage, that means the correct seeding order is:
> **Home → Case Index → Source Page → Claim Page.**

That is the current seeding-order ruling.
