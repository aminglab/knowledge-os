# home-seed-surface-v1

## Status

Seed-surface note v1.

## Function

This document defines the first **Home seed surface** for the public layer of Knowledge OS.

It follows the current Phase 3 chain:

- `phase-3-direction-decision-v1.md`
- `public-layer-foundation-v1.md`
- `public-layer-ia-v1.md`
- `public-layer-seeding-order-v1.md`

That chain already decided three things:

1. Phase 3 is a **public-layer foundation phase**.
2. The public layer now exists in seed form.
3. The next seeding order begins with **Home**.

This document therefore answers a narrower question:

> What is the smallest honest Home surface that Knowledge OS should expose as a public entry right now?

This is not the final public homepage specification.
It is the minimum credible seed for a system-level public entry.

---

## Why Home must be seeded now

At the current repository stage, the public layer already has a real first case surface.
That is good.
But it also creates a structural imbalance.

Right now, the first case is doing too much work.
It is functioning as:

- the first public case,
- the first public page,
- and part of the system’s public identity surface.

That is acceptable as a transitional condition.
It is not a good long-term public-layer structure.

A system-level public entry must therefore exist so that a reader can answer, before entering a case:

- what this system is,
- what kind of public artifact it exposes,
- why the public layer exists,
- and where the first public case fits inside a larger whole.

So the Home seed is not cosmetic.
It is the first structural separation between:

- **system-level public entry**, and
- **case-level public entry**.

---

## Core definition

The **Home seed** is the first public surface that introduces Knowledge OS as a system before a reader enters any one case.

Its purpose is not to explain everything.
Its purpose is to orient the reader correctly.

A short definition is:

> Home is the smallest public entry surface that tells a reader what Knowledge OS is, why the public layer exists, and where to begin.

---

## What Home should accomplish

The Home seed should accomplish five things and no more.

### 1. Establish system identity

A reader should immediately understand that they are entering:

- not a generic article site,
- not a chat product,
- not a static archive,
- but a public layer over governed knowledge objects.

### 2. Explain the public-layer premise

The page should make legible the key shift:

- static publication artifact,
- versus living governed knowledge surface.

This should be visible without forcing the reader to read the full founding note first.

### 3. Introduce the first public case as a case, not as the whole system

The first case should be featured clearly.
But it should appear as:

- the first public case,
- the first seed implementation,
- not the entirety of Knowledge OS.

### 4. Provide a clean first route forward

A new reader should know what to do next.
At minimum, Home should route to:

- the first public case,
- and the future Case Index surface.

### 5. Stay honest about stage

The Home seed should not pretend the whole public product is already built.
It should present:

- a real public entry,
- a real first case,
- and a real public-layer direction,
- while remaining explicit that this is an early but genuine foundation stage.

---

## What Home should not try to do

The Home seed should stay small on purpose.
It should not try to become any of the following.

### 1. Not a full marketing site

This is not the phase for a broad promotional website with many top-level narratives.

### 2. Not a full public dashboard

The Home seed should not pretend that the public layer already has dozens of surfaces, metrics, filters, or live public participation systems.

### 3. Not a clone of README.md

The repository README is a repository entry document.
The Home seed is a public-layer entry surface.
Their roles overlap, but they are not identical.

### 4. Not a replacement for the first case page

Home should hand off to the first case.
It should not absorb the case into itself.

### 5. Not a theory dump

The Home seed should not front-load too much architecture prose, protocol detail, or developer-facing governance language.

---

## Minimum Home surface structure

The Home seed should contain six blocks.

### Block 1. System identity block

Purpose:

- establish what Knowledge OS is in one screen.

Minimum contents:

- system name,
- one-sentence definition,
- short clarifying sentence about what it is not,
- and a concise statement that this is a public layer over governed knowledge objects.

Suggested content shape:

- **Knowledge OS**
- one-sentence definition
- “not a chat wrapper / not an automated paper factory / not a static archive”

This block should feel more like a product identity surface than a README heading.

---

### Block 2. Why this exists block

Purpose:

- explain the structural problem the system is attacking.

Minimum contents:

- dynamic knowledge vs static publication,
- short problem framing,
- short statement of what changes in a living governance model.

This block should not expand into the full Founding Note.
It should only give a public reader enough context to understand why the public layer exists.

---

### Block 3. Public-layer explanation block

Purpose:

- explain what kind of public artifact the reader is about to enter.

Minimum contents:

- this system publishes governed release views,
- the public layer is selective,
- a case page is a living public case, not just an article,
- and the reader can move from summary into claim, source, and lineage surfaces.

This is one of the most important blocks, because it prevents public misunderstanding of what the first case page is.

---

### Block 4. First case feature block

Purpose:

- feature the first public case clearly and honestly.

Minimum contents:

- case title,
- one-paragraph synopsis,
- current visible judgment summary,
- why this case matters,
- and link into the case page.

This block should make the first case feel important without making it appear to be the whole public layer.

---

### Block 5. Public entry routes block

Purpose:

- tell the reader where to go next.

Minimum contents:

- enter first case,
- future route to case index,
- optionally a short “how to read this system” route.

At the current stage, there may only be one live forward route.
That is fine.
The important point is to expose navigation intentionally.

---

### Block 6. Current-stage honesty block

Purpose:

- keep the public layer honest about maturity.

Minimum contents:

- this is an early public-layer foundation,
- the first case is the first seed implementation,
- the system is not yet the full operating environment in public form.

This block protects against overclaim while still presenting the work as real.

---

## Minimum copy posture

The Home seed should follow a copy discipline that is different from both:

- repository README prose,
- and case-page prose.

### It should be:

- clear,
- direct,
- public-facing,
- product-aware,
- and modestly staged.

### It should not be:

- overly internal,
- overly technical,
- over-theoretical,
- or startup-marketing-flavored in a generic way.

The reader should come away thinking:

- “I understand what this system is trying to do,”
- “I understand why this first case exists,”
- and “I know where to click next.”

## Home seed versus repository README

This distinction matters enough to state explicitly.

### README

The repository README is:

- repository-facing,
- contributor-friendly,
- and still partly shaped for code / project entry.

### Home seed

The Home seed should be:

- public-layer-facing,
- reader-facing,
- and shaped as the first product-like public entry.

They can share language.
They should not be treated as the same artifact.

---

## Current implementation stance

The Home seed does not need a full new frontend stack decision before it can be defined.

At the current stage, the most honest implementation stance is:

- define the Home seed surface first,
- keep it light,
- reuse existing project language where valid,
- and avoid turning Home into a heavy app shell prematurely.

This means the next implementation-minded step after this document is likely not a full new system build.
It is a light seed pass that clarifies:

- what Home must say,
- what Home must link to,
- and how Home visually differs from the case page.

---

## Engineering consequence

From a project-engineering perspective, the Home seed does three useful things.

### 1. It reduces role confusion

Once Home exists, the first case page no longer has to carry system-level identity by itself.

### 2. It creates a stable top-level target for later Case Index insertion

Case Index can later slot naturally under Home rather than being awkwardly attached straight to repository docs.

### 3. It gives public-layer implementation a cleaner boundary

Instead of “keep making the first case page smarter,” the project gets a cleaner rule:

- system identity belongs to Home,
- case identity belongs to Case Page,
- listing belongs to Case Index,
- traceability belongs to Source and Claim surfaces.

That is exactly the kind of boundary that prevents later public-layer drift.

Current verdict:

>
> It should be the smallest honest public entry that establishes system identity, explains the public-layer premise, features the first case as a first case, and routes the reader forward without overclaiming product maturity.

That is the current Home seed ruling.
