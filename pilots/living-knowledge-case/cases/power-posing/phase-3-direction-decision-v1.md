# phase-3-direction-decision-v1

## Status

Phase direction decision v1.

## Function

This document records the formal strategic direction decision for **Phase 3** of the `power-posing` line.

It does not reopen the question of whether the first line is real.
That question has already been settled by the completed Phase 0–2 chain.

Its purpose is narrower and harder:

- to record what the project has already achieved,
- to define what Phase 3 is actually for,
- to select the primary strategic direction,
- to state what this phase will explicitly not do,
- and to define the next concrete drafting sequence.

---

## Current stage judgment

The project is no longer in the state of:

- white paper only,
- repository constitution only,
- case prototype only,
- or page experiment only.

It has already become a minimum but real three-layer system:

- a stable project entry on `main`,
- a real living knowledge case centered on `power-posing`,
- and a first public-facing case page as downstream release surface.

This means the line no longer needs existential self-justification.
The next task is no longer to prove that the line exists.
The next task is to decide which larger line should grow out of it.

So the correct starting sentence for Phase 3 is:

> The first line has already been made real. Phase 3 is a direction choice, not an existence proof.

---

## Decision frame

At the end of Phase 2, four broad strategic directions were visible.

### A. Public-layer / product phase

Treat the first case surface as established, and move from a single case page toward a clearer public-layer product structure.

### B. Protocol / object-side broader lifting

Return attention to object-envelope, checks, grammar, and governance surfaces, and decide whether any of them should be lifted from case-earned discipline into broader pilot discipline.

### C. Second-case launch

Open a second case in order to test seam portability and replication.

### D. Stabilize without expansion

Pause major expansion, keep the achieved line stable, and defer larger construction.

The question for Phase 3 is not whether all four are meaningful.
They are.
The question is which one is most valuable **now**.

---

## Formal decision

### Phase 3 decision

**Phase 3 will enter Direction A:**

> **public-layer foundation phase**

But this must be read carefully.

This is **not** a decision to continue polishing the current `power-posing` page indefinitely.

It is a decision to use the already-established first public case surface as the seed implementation for a more explicit **public-layer foundation**.

In other words:

- `power-posing` remains the first validated case,
- the page remains the first validated public surface,
- but Phase 3 shifts the center of gravity away from local page polishing,
- and toward the product/system meaning of the public layer itself.

So the actual Phase 3 verdict is:

> **Do not continue local page polishing as the main line.**
> **Do not open a second case yet.**
> **Do not prematurely generalize protocol into broad doctrine.**
> **Use the first validated case surface to define the first public-layer foundation.**

---

## Why Direction A is selected now

### 1. Because Phase 2 already pushed the first page past the key threshold

The first public case page has already crossed the important line:

- it is no longer structurally unstable,
- it is no longer mainly a narrative prototype,
- and it is no longer waiting for existence proof.

Further work on the same page is now more likely to produce diminishing returns.

So Phase 3 should not be framed as:

- make the page slightly nicer,
- tune the wording again,
- or add another local patch.

That would confuse tactical finishing with strategic progress.

### 2. Because broader lifting is still too early as the central move

Phase 3 is not the right moment to over-promote case-earned governance surfaces into general doctrine too quickly.

Object-envelope enforcement, check grammar, and governance layers are already valuable.
But they have still been earned primarily inside one case.

If broader lifting becomes the main line too early, the project risks generating abstract protocol inflation before enough public-layer pressure has been absorbed.

So Direction B remains legitimate, but not yet primary.

### 3. Because the project now needs a product-level reading of what it has built

The most valuable next move is to reinterpret the achieved first case page not as an isolated success, but as the first visible node in a larger product/public-layer architecture.

That architecture has not yet been properly objectified.
Phase 3 is the correct place to do that.

### 4. Because opening a second case now would be strategically noisy

A second case would add pressure, but it would also split attention.
At the current stage, the more valuable move is not another sample.
It is to define the foundation that tells us what the public layer actually is.

So the project should resist the temptation to use “more cases” as a substitute for strategic clarity.

---

## What Phase 3 is actually for

Phase 3 is for one task:

> to convert “the first public case surface exists” into “the project has a defined public-layer foundation.”

That means Phase 3 should answer questions like:

- What is the public layer for?
- What kinds of objects belong to it?
- What is the minimum page universe of the public layer?
- What is the relation between case page, claim page, source page, snapshot, and release identity?
- What remains case-scoped, and what begins to become system-level?
- What is the difference between a public release surface and a full cockpit/workspace?
- How should the first case be situated inside the larger Knowledge OS identity?

Phase 3 is therefore not just a UI pass.
It is a product-meaning pass.

---

## Core Phase 3 principle

The central principle of this phase is:

> **From first page to first public-layer foundation.**

This principle has four implications.

### 1. `power-posing` remains seed, not center of the universe

The project should not pretend `power-posing` is the final public-layer form.
It is the first validated seed.

### 2. The page should now be read as seed implementation

The current page is not merely a page.
It is the first seed implementation of the public layer.

### 3. Snapshot remains the upstream public release layer

The discipline already established in earlier phases must remain intact:

- object layer is deepest source,
- `snapshot-v2.md` is the current public release layer,
- page is downstream presentation.

Phase 3 must grow on top of that rule, not violate it.

### 4. Public-layer meaning must be clarified before heavy product build-out

The project should not jump straight into richer interaction shell, graph systems, or multi-case product expansion without first clarifying what the public layer is structurally supposed to contain.

---

## Explicit non-goals for Phase 3

Phase 3 will **not** treat the following as its main line.

### 1. Not indefinite local polishing of the current page

Small fixes may still happen.
But they are not the phase center.

### 2. Not second-case launch

A second case remains deferred.

### 3. Not full generic multi-case architecture

The project should not pretend it is ready for a full generic public-layer engine.
This phase is about foundation, not premature system completion.

### 4. Not major protocol constitutional expansion

This is not the phase for writing a vast new doctrine layer.
Protocol lifting should remain case-earned and selective.

### 5. Not a full cockpit build

Phase 3 is not yet the full private cockpit / team workspace construction phase.
It is specifically about the public-layer side.

---

## Minimum deliverables for Phase 3

The phase should count as successful only if it produces at least the following foundation artifacts.

### 1. Public-layer foundation document

A formal statement of what the public layer is, what it contains, and what role it plays in the whole system.

Suggested file:

- `public-layer-foundation-v1.md`

### 2. Public-layer information architecture document

A concrete IA description of the minimum public-layer page universe.

Suggested file:

- `public-layer-ia-v1.md`

### 3. System placement note for the first case

A document that explains how `power-posing` functions as:

- first living case,
- first public case surface,
- seed implementation of the public layer,
- not-yet-generic but no longer merely local experiment.

This can be folded into one of the above, or broken out separately if necessary.

### 4. One explicit public identity pass

The public layer should begin to state more clearly:

- what Knowledge OS is,
- why this page exists,
- how this page relates to the larger system,
- and what the reader is seeing: a case page, not the whole operating environment.

---

## Minimum page universe to define during Phase 3

Phase 3 should define a minimum public-layer page universe with at least the following objects.

### 1. Home

The project’s public entry point.
Not a full cockpit.
Not merely a manifesto.
A public entry into the living knowledge system.

### 2. Case Index

A public view of available or published cases.
Even if only one case exists now, the page type should be defined.

### 3. Case Page

The current `power-posing` page is the first seed of this page type.

### 4. Claim Page

A more atomic public page type for claim-centered navigation.
This may remain not-yet-implemented, but should be defined.

### 5. Source Page

A public-facing source/surface page type, distinct from raw metadata storage.

### 6. Reading / lineage layer

A way to show how cases, claims, sources, and release surfaces relate to one another over time.

Phase 3 does not need to fully build all of these.
But it should define them clearly enough that later product work is not blind.

---

## Recommended work sequence

### Step 1. Adopt this direction decision

First, the project must formally adopt the Phase 3 direction decision.
Without this, later work risks slipping back into local page tinkering.

### Step 2. Draft `public-layer-foundation-v1.md`

This should define:

- why the public layer exists,
- what problems it solves,
- what it is not,
- how it differs from private cockpit and team layer,
- what object/release identity it exposes.

### Step 3. Draft `public-layer-ia-v1.md`

This should define:

- minimum public page types,
- expected reader journeys,
- system identity cues,
- relationship between case surface and larger Knowledge OS surface.

### Step 4. Decide whether one light public identity implementation pass is warranted

Only after the foundation and IA are written should the project decide whether a light implementation pass is needed in the current page, home surface, or public entry.

### Step 5. Reassess whether Phase 4 should remain on public-layer expansion or return toward protocol-side lifting

Phase 3 should end with a new strategic checkpoint.
It should not silently roll into another unbounded phase.

---

## Risk note

The biggest risk in Phase 3 is not technical failure.
It is directional drift.

That drift could take three forms.

### 1. Falling back into local page polishing

This would consume attention without changing project-level shape.

### 2. Jumping too early into heavy product build-out

This would create visible structures before the public-layer meaning has been clarified.

### 3. Retreating into protocol abstraction

This would produce documents that feel rigorous but are not yet earned by enough public-layer pressure.

The function of this decision document is precisely to keep the project out of those three traps.

---

## Final verdict

The first line has already been made real.
The first public case surface has already been made real.

Therefore the correct next move is not:

- more existence proof,
- more local polishing,
- or more premature generality.

The correct next move is:

> **to establish the first public-layer foundation of Knowledge OS, using the already-validated first case surface as seed implementation.**

That is the formal direction decision for Phase 3.

---

## Consolidated closing sentence

Phase 3 is not about continuing to polish `power-posing` locally.
It is about turning the first validated public case surface into the first defined public-layer foundation of Knowledge OS.
