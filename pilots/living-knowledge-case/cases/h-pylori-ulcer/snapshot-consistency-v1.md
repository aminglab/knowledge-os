# Snapshot Consistency v1

This note records the current **snapshot consistency rule v1** for the `h-pylori-ulcer` case.

Its purpose is narrow:

- keep `snapshot-v0.md` from drifting away from the current verdict layer,
- keep its object counts and reading routes aligned with the current case layer,
- keep its honesty note aligned with the real state of the current source floor,
- and give the snapshot layer a checker-friendly consistency surface.

This note is case-scoped.
It does **not** claim to be a repository-wide snapshot doctrine.

---

## Why this layer exists

The current snapshot stack already protects two things:

- section structure
- subsection semantic anchors

That is necessary.
It is not sufficient.

A snapshot can keep the right headings and the right local semantic phrases while still drifting away from:

- the current verdict levels,
- the current object totals,
- the current reading routes,
- or the real current state of the source floor.

This file exists to stop that drift.

---

## One-sentence ruling

Current ruling:

> **The snapshot should remain consistent with the current verdict layer, current object totals, current reader routes, and current source-floor reality.**

Do not let `snapshot-v0.md` become a structurally elegant but case-inconsistent page.

---

At the current stage, `snapshot-v0.md` should satisfy the following.

### 1. Current visible judgment alignment

The `Current visible judgment` section should remain aligned with the current verdict layer:

- `V-0001` for the stronger claim
- `V-0002` for the narrower descendant claim

At the current stage, that means the snapshot should continue to surface:

- `stabilized` for the stronger claim
- `supported` for the narrower descendant claim

### 2. Core claim neighborhood alignment

The `Core claim neighborhood` section should continue to route readers to the current public claim pages for:

- `C-0001`
- `C-0002`

### 3. Included object counts

The `Included objects` section should continue to reflect the current governed object totals for:

- claims
- evidence objects
- dissent objects
- verdict objects

### 4. Reading path reality

The `Snapshot reading path` section should continue to route to the current public entry surfaces that actually exist.

### 5. Source-floor honesty

The `Current-stage honesty note` should continue to be true when it says the snapshot sits above:

- a stable metadata floor,
- a source-page index,
- and first written source pages for the canonical case sources.

That means the current source-page set and current canonical source-id set should still match the snapshot’s honesty claim.

---

## What this note does **not** require

This note does **not** require that the snapshot:

- become a full graph explorer,
- list every object id in the whole case,
- mention every source id inline,
- or behave like a page-emission product.

The goal is not maximal density.
The goal is honest case consistency.
