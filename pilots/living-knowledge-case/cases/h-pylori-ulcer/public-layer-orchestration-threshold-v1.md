# Public-Layer Orchestration Threshold v1

This note records the current **public-layer orchestration threshold v1** for the `h-pylori-ulcer` case.

Its purpose is narrow:

- translate the current do-not-expand-yet stance into threshold language,
- say what would justify a later orchestration expansion,
- and stop the next expansion from being triggered by restlessness alone.

This note is case-scoped.
It does **not** claim to define a repository-wide orchestration threshold.

---

## One-sentence ruling

Current ruling:

> **Keep the current orchestration boundary as-is for now.**
> **Do not open a larger orchestration expansion until a real forcing condition appears.**

---

## Current retained hold

The current stage continues to retain:

- `HOLD_NO_LARGER_PUBLIC_LAYER_ORCHESTRATION_BOUNDARY`

That hold is still correct.
The current boundary is now named and governed, but it is not yet forced to expand further.

---

## What would justify a later expansion

A larger orchestration expansion becomes justified when one or more of the following becomes true.

### Trigger 1. More than one operational public-layer entrypoint becomes necessary

For example:

- a page-emission validator joins the current stack,
- a second workflow or Make target becomes first-class,
- or a hidden-checks / visible-checks split becomes operationally necessary.

### Trigger 2. The current suite entrypoint becomes too coarse to localize orchestration failures cleanly

If the current `check_h_pylori_public_layer.py` surface becomes too blunt to explain where orchestration-level failure now lives, that is a real signal that the boundary has outgrown its current shape.

### Trigger 3. The current atlas / threshold / governance surfaces begin to multiply again

If the current boundary note, threshold note, atlas, and atlas-governance self-check are no longer enough to keep the orchestration layer legible, then the next level of orchestration structure becomes much easier to justify.

### Trigger 4. The second case gains a page-emission layer or first-case-equivalent public release surface

Once the second case has a larger public-facing release machinery, orchestration can no longer remain only a suite-and-workflow story.

---

## What should not count as a trigger

The following should **not** count as sufficient triggers by themselves:

- boredom with the current note set,
- desire for symmetry with the first case,
- imagined future complexity,
- or the vague feeling that the stack "looks mature enough."

The threshold should remain pressure-driven rather than aesthetics-driven.

---

## Practical verdict

So the practical verdict is:

- the current orchestration boundary is now explicit,
- the larger expansion question is now a threshold question rather than a mood,
- and the current hold remains lawful until the stack produces a genuine forcing condition.
