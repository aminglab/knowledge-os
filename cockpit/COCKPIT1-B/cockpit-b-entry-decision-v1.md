# COCKPIT1-B Entry Decision v1

## Status

Entered at current threshold.

## Function

This file records the entry decision for **COCKPIT1-B / Minimal Static Cockpit Prototype v1**.

COCKPIT1-A bound the first cockpit object space to real governed cases.
The next lawful step is to materialize those bindings into a low-fidelity static frontend skeleton.

---

## One-sentence entry verdict

Current entry verdict:

> **ENTER_COCKPIT1_B_STATIC_PROTOTYPE**
>
> The project may now build a zero-build static cockpit prototype because the cockpit grammar, case bindings, and Phase 1 demo scope have already been formally bounded.

---

## Immediate purpose

COCKPIT1-B exists to prove four things:

1. the cockpit can be rendered as a real object-centered screen rather than only described in prose;
2. the five required screen states can be shown through a single bounded prototype;
3. the two currently bound cases can coexist inside one cockpit frame without collapsing into one generic example;
4. the sidecar can remain visibly subordinate while real case objects stay in the center.

---

## Boundary

COCKPIT1-B does **not** yet:

- implement live write-capable runtime logic
- route real model calls
- replace the public-layer pages
- open team-layer interaction
- claim production-ready frontend behavior

It is a static skeleton only.

---

## Final decision block

- entry verdict: `ENTER_COCKPIT1_B_STATIC_PROTOTYPE`
- prototype kind: `zero-build static cockpit skeleton`
- bound cases: `power-posing`, `h-pylori-ulcer`
- proof target: `five required cockpit screen states`
