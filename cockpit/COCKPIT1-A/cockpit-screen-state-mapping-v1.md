# COCKPIT1-A Screen State Mapping v1

## Status

Drafted for mapping.

## Function

This file defines the minimum cockpit screen states that must be demonstrated with real case bindings before implementation-grade claims become lawful.

---

## Required screen states

### 1. Project landing state
Must show:

- bound case selection
- case summary pointer
- route or stage entry choice

### 2. Claim focus state
Must show:

- active claim title
- current verdict / judgment
- direct support and challenge surfaces
- next actions

### 3. Dissent pressure state
Must show:

- current dissent queue
- unresolved pressure
- candidate response actions

### 4. Verdict boundary state
Must show:

- current verdict token
- current holds
- current non-upgrade boundary
- what would be needed for lawful movement

### 5. Route / map state
Must show:

- case route tree or equivalent project map
- current focus pointer
- navigable object entry points

---

## Case mapping

Recommended demonstration order:

1. `h-pylori-ulcer` claim focus state
2. `h-pylori-ulcer` verdict boundary state
3. `power-posing` dissent pressure state
4. `power-posing` route / map state
5. shared project landing state

This order favors current governance clarity first, then richer route complexity.

---

## Final mapping block

- required screen states: 5
- minimum real-case bindings per state: 1
- implementation claim before state mapping: forbidden
